# Codex Evidence Lab 服务套件设计

日期：2026-07-19
目标 Agent：OKX.AI ASP #3359 `Codex Evidence Lab`

## 1. 背景

#3359 当前登记了 3 项服务：1 项 agent to agent 定制研究和 2 项 API service。Agent 因平台功能验证消息未收到回复而被拒绝上架；与此同时，市场页的服务颗粒度偏粗，未完整呈现已经部署的市场数据与交易前风控能力。

本次改造保留 #3359，不新建 Agent，也不修改已上线的 #4515 `Market Signal Desk`。目标是把 #3359 重构为边界清楚、按次调用、可验证的链上代币风控套件，并保留一项多轮定制研究服务。

## 2. 目标与非目标

### 目标

- 在同一套 Zeabur 运行时中提供 8 个可独立调用的 HTTPS API service。
- 保留 1 个 agent to agent 定制研究服务，最终市场页共展示 9 张服务卡。
- 每个 API 使用独立输入、输出和风险边界，避免同一结果仅更换名称。
- 所有付费接口复用现有 OKX x402 收款配置。
- 平台功能验证消息先快速确认收到，再进入完整处理，降低审核超时概率。
- 数据不可用时显式返回未知或部分覆盖，不推断或编造安全结论。

### 非目标

- 不执行交易、授权、签名、转账或自动下单。
- 不承诺收益或给出确定性买卖结论。
- 不把仅支持 EVM 的合约安全字段宣称为全链覆盖。
- 不为增加服务数量而复制同一输出。

## 3. Agent 定位

名称保持：`Codex Evidence Lab`

建议简介：

> 面向交易 Agent 与研究场景的链上代币风控服务，提供市场、流动性、成交、新币启动、量价异常、合约权限、交易税、持仓集中度与综合交易前风险检查。结果包含结构化指标、风险分级、数据完整度和可核验来源，缺失数据明确标记未知。

## 4. 服务目录

### 4.1 API service

#### 1. 代币市场快照

- 价格：`0.01 USDT`
- 接口：`POST /api/token-market-snapshot`
- 核心能力：汇总代币价格、市值、FDV、主要交易池、流动性与多周期涨跌，返回结构化市场快照、数据完整度和来源。
- 用户需提供：网络和代币合约或铸币地址，可选输出语言。

#### 2. 流动性风险扫描

- 价格：`0.01 USDT`
- 接口：`POST /api/liquidity-risk-scan`
- 核心能力：比较主要交易池流动性、池间分布、流动性与市值比例，返回流动性风险等级、异常标记和来源。
- 用户需提供：网络和代币合约或铸币地址，可选最低流动性阈值。

#### 3. 成交活跃度分析

- 价格：`0.01 USDT`
- 接口：`POST /api/trading-activity-scan`
- 核心能力：分析多周期成交量、买卖次数、买卖方向和活跃度变化，返回成交活跃度等级与异常信号。
- 用户需提供：网络和代币合约或铸币地址，可选观察周期。

#### 4. 新币启动风险检查

- 价格：`0.01 USDT`
- 接口：`POST /api/new-pair-risk-check`
- 核心能力：结合交易对年龄、初始流动性、成交与价格变化，识别新币启动阶段的流动性和量价风险。
- 用户需提供：网络和代币合约或铸币地址，可选风险偏好。

#### 5. 价格成交异常扫描

- 价格：`0.01 USDT`
- 接口：`POST /api/market-anomaly-scan`
- 核心能力：检测多周期价格、成交量和买卖次数变化，标记急涨急跌、量价背离和交易活动异常。
- 用户需提供：网络和代币合约或铸币地址，可选观察周期与异常阈值。

#### 6. 合约权限与交易税检查

- 价格：`0.02 USDT`
- 接口：`POST /api/contract-tax-check`
- 核心能力：检查合约开源、代理、增发、暂停、黑名单、蜜罐及买卖税等安全项，返回字段状态和风险分级。
- 用户需提供：受支持的 EVM 网络和代币合约地址。

#### 7. 持仓集中度检查

- 价格：`0.02 USDT`
- 接口：`POST /api/holder-concentration-check`
- 核心能力：分析持有人数量、前十大持仓比例、创建者与所有者持仓及流动性持有情况，返回集中度风险。
- 用户需提供：受支持的 EVM 网络和代币合约地址。

#### 8. 综合交易前风险报告

- 价格：`0.03 USDT`
- 接口：`POST /api/pretrade-risk-report`
- 核心能力：整合市场、流动性、成交、新币阶段、量价异常、合约权限、交易税与持仓风险，返回综合评分和分项证据。
- 用户需提供：网络和代币地址；EVM 返回完整安全项，其他网络按可用数据返回并标记覆盖范围。

### 4.2 Agent to agent

#### 9. 定制代币风险研究

- 价格：协商
- 核心能力：按需完成代币公开数据核查、市场与安全风险整理、来源交叉验证和结构化研究报告。
- 用户需提供：网络、代币地址、关注问题、交付语言、截止时间和验收标准。

## 5. 数据架构

### DexScreener

作为市场数据主来源，提供价格、FDV、市值、流动性、成交量、买卖次数、多周期价格变化和交易对创建时间。市场类服务支持现有链映射：Solana、Ethereum、X Layer、Base、BSC、Arbitrum 和 Polygon。

### GoPlus

作为 EVM 安全数据来源，提供合约开源、代理、增发、暂停、黑名单、交易税、蜜罐、持有人数量、前十大持仓及创建者/所有者持仓等字段。无法识别或未返回的字段保持 `unknown`，不得推断。

### GeckoTerminal

作为市场数据复核和备用来源。仅在主来源缺失或需要交叉验证时调用，避免其公共接口限流影响主链路。

### 缓存与超时

- 市场数据缓存：30 秒。
- 安全数据缓存：5 分钟。
- 单个上游请求设置超时；综合报告允许部分成功，但必须返回 `data_quality` 和缺失来源。
- 缓存键包含网络、代币地址和服务所需参数。

## 6. 接口合同

所有接口接受 JSON，请求字段以 `chain`、`token_address` 和 `language` 为基础。服务特有参数采用可选字段，例如 `lookback_hours`、`min_liquidity_usd`、`risk_profile` 和 `anomaly_threshold`。

所有成功响应至少包含：

- `ok`
- `service`
- `version`
- `request_id`
- `generated_at`
- `input`
- 服务专属结果
- `risk_level` 或等价分级
- `flags`
- `data_quality`
- `sources`

错误响应采用稳定的错误码，区分输入无效、链不支持、代币不存在、上游限流、上游超时和内部错误。错误中不返回密钥、签名、付款凭证或内部堆栈。

## 7. 兼容与迁移

- 现有 `/api/token-risk-scan` 保留，作为 `/api/pretrade-risk-report` 的兼容别名。
- 现有 `/api/ape-pretrade-check` 保留，作为 `/api/new-pair-risk-check` 的兼容别名。
- 现有 `/api/signal-snapshot` 保留，并映射到新版市场快照能力。
- MCP 工具路由继续暴露现有工具，并新增 8 个明确命名的工具。

市场登记迁移：

- 服务 `33822`：`Token Risk Review` 更新为“定制代币风险研究”。
- 服务 `33823`：`Token Risk Guard` 更新为“综合交易前风险报告”。
- 服务 `33824`：`ApeGuard` 更新为“新币启动风险检查”。
- 新增其余 6 个 API service。

## 8. A2A 在线与快速响应

A2A 通信进程继续部署在 Zeabur 云服务器，独立于 API 服务运行。收到平台功能验证或用户消息后：

1. 监听器立即发送简短确认，表明 Agent 在线且已收到消息。
2. 后台再进行完整任务解析和处理。
3. 记录接收时间、确认时间、会话标识和最终处理状态，且不记录敏感凭证。
4. 进程异常退出后由服务管理器自动重启，并保留健康检查。

## 9. x402 与安全

- 8 个付费接口复用现有 OKX facilitator 和 X Layer USDT0 收款地址。
- 未付款请求返回标准 HTTP 402、完整公开 HTTPS 资源地址和正确金额。
- 凭证仅通过 Zeabur 环境变量注入，不写入仓库、日志或接口响应。
- `.env` 保持在 `.gitignore` 中。
- 本次服务均为只读分析，不要求用户提供私钥、助记词或交易签名。

## 10. 验收标准

重新提交 #3359 前必须满足：

- 8 个新版接口在本地测试和 Zeabur 公网环境中均可访问。
- 每个接口的输入校验、正常输出、上游缺失和上游超时路径均有测试。
- 8 个公开接口的未付款请求均返回有效 x402 挑战和完整 HTTPS URL。
- `/health`、`/metadata`、`/openapi.json` 和 MCP 工具列表与服务目录一致。
- 兼容接口继续通过原有烟雾测试。
- A2A daemon 健康检查无阻断项，云端监听器可快速确认测试消息。
- 使用测试用户调用 #3359，确认平台能收到 Agent 回复。
- 上架描述逐项与实际输出核对，未实现字段不得写入服务文案。

## 11. 上架流程

1. 完成代码与自动化测试。
2. 部署到 Zeabur 并验证公网接口。
3. 进行 A2A 手动往返测试。
4. 更新 #3359 的简介和现有 3 项服务，新增 6 项服务。
5. 展示更新前后差异及最终提交卡，等待用户明确确认。
6. 保存更新后重新激活 #3359，发起上架审核。
